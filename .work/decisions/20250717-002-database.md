# ADR-002: PostgreSQL + pgvector database

**Status:** Decided · **Date:** 2025-07-17

## Context

RFP Platform stores structured data (users, RFPs, questions, drafts, approvals) and vector embeddings (knowledge base chunks). Running two separate databases (relational + vector) increases ops surface — especially important given the solo/small-team constraint.

## Decision

Use **PostgreSQL 16 with pgvector extension** as the single database for both relational data and vector embeddings.

## Consequences

- **Positive:** Single database to manage, back up, and tune; pgvector provides good-enough ANN search for V1 scale; no additional operational cost.
- **Negative:** pgvector may lag behind dedicated vector DBs for >1M embeddings or sub-10ms latency requirements; V1 volume (10s of RFPs, ~100K chunks) well within pgvector's sweet spot.
- **Neutral:** Migration from pgvector to a dedicated vector DB (Pinecone, Qdrant) is possible post-V1 if scale demands it.

## Alternatives considered

| Alternative | Reason rejected |
|-------------|-----------------|
| PostgreSQL + separate Pinecone/Qdrant | Increased ops surface, cost, and latency for V1 volumes |
| SQLite + pgvector fork | Weaker concurrency model for multi-tenant SaaS |
| MongoDB Atlas + vector search | Vendor lock-in; contradicts self-hosted requirement |
