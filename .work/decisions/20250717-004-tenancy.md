# ADR-004: Multi-tenant architecture with RLS

**Status:** Decided · **Date:** 2025-07-17

## Context

The platform serves multiple customer organizations (tenants) from a single deployment. Each tenant's data (RFPs, knowledge base, drafts, users) must be strictly isolated. V1 targets a self-hosted Docker Compose deployment with solo/small-team operations.

## Decision

Use **single-database multi-tenant with Row-Level Security (RLS)** on PostgreSQL.

## Consequences

- **Positive:** Single database = single backup, single migration target, simpler ops; RLS provides database-enforced isolation at the row level — no risk of application-level leakage; new tenants are zero-config (just create an organization and tenant_id).
- **Negative:** Noisy-neighbor problem possible (one tenant's heavy query affects others — mitigated by PostgreSQL resource limits at V2); tenant data can't be physically separated without migration.
- **Neutral:** Every table includes `tenant_id`; all queries filter through RLS policies; application code never explicitly filters by tenant (RLS does it transparently).

## Alternatives considered

| Alternative | Reason rejected |
|-------------|-----------------|
| Schema-per-tenant | More complex migrations (N schemas); harder to manage with solo team |
| Single-tenant per deployment | Higher ops burden for multi-tenant product; contradicts SaaS delivery model |
| Database-per-tenant | Prohibitively expensive at small scale; connection management overhead |
