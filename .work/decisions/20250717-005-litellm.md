# ADR-005: LiteLLM multi-provider LLM adapter

**Status:** Decided · **Date:** 2025-07-17

## Context

AI draft generation requires access to LLMs. Vendor lock-in to a single provider (OpenAI, Anthropic, Google) creates risk around pricing changes, availability, and data privacy. Different RFP content may benefit from different models (cheaper for parsing, more capable for generation).

## Decision

Use **LiteLLM** as the multi-provider abstraction layer for all LLM calls.

## Consequences

- **Positive:** Single API interface for all providers; built-in support for fallback chains, cost tracking, rate limiting, and retries; easy to add new providers without code changes; non-training headers supported (data privacy).
- **Negative:** Slight latency overhead from the abstraction layer (configurable timeout per provider); potential dependency on LiteLLM project maintenance.
- **Neutral:** Configuration-driven provider selection per tenant via Admin UI.

## Alternatives considered

| Alternative | Reason rejected |
|-------------|-----------------|
| Direct provider SDKs | Tight coupling; multi-provider fallback requires custom code |
| LangChain | Heavier framework with broader scope than needed (agent frameworks, chains, tools); higher latency and complexity for simple RAG + generation |
| Custom abstraction | Build vs. buy — not worth the engineering time for solo team |
