# ADR-006: Docker Compose deployment

**Status:** Decided · **Date:** 2025-07-17

## Context

The platform must be self-hosted (customer requirement from probe), with a solo/small team operating it. All services (FastAPI, Celery, PostgreSQL, Redis, MinIO, frontend) must run on a single machine with minimal operational overhead.

## Decision

Use **Docker Compose** for local development and production deployment.

## Consequences

- **Positive:** Single `docker compose up` for full stack; reproducible environments across dev, staging, and production; easy for customers to self-host; no cloud vendor dependency.
- **Negative:** Single-node deployment limits horizontal scaling (post-V1 concern); no built-in orchestration for multi-node; no zero-downtime deploy out of the box.
- **Neutral:** Migrating to Kubernetes or a PaaS is possible post-V1 if scale demands it; documented in migration path.

## Alternatives considered

| Alternative | Reason rejected |
|-------------|-----------------|
| Kubernetes | Overkill for solo team and V1 scale; high ops overhead |
| AWS ECS + RDS | Contradicts self-hosted requirement; vendor lock-in |
| Bare metal / shell scripts | Reproducibility issues; higher maintenance burden |
