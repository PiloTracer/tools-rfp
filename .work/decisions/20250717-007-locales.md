# ADR-007: UI/Document locales

**Status:** Decided · **Date:** 2025-07-17

## Context

RFP Platform serves B2B teams that may operate in multilingual environments (e.g., Canadian or European markets with bilingual RFP requirements). V1 needs to support output in languages other than English for response generation.

## Decision

Support **English and Spanish** in V1 UI and DOCX exports. Implement with a flat i18n locale file structure (JSON) and a React i18n library (react-i18next). Additional languages can be added by contributing locale JSON files.

## Consequences

- **Positive:** Covers the most common bilingual RFP scenario (US + Latin American markets); flat locale files are easy for non-developers to translate; template-based DOCX export can switch language context.
- **Negative:** Adds ~30–50% overhead to UI development (every string must use `t()` wrapper); RFP parsing remains language-agnostic (parsers work on content, not UI).
- **Neutral:** Spanish can be added incrementally — English-first V1 with Spanish locale files as translation target.

## Alternatives considered

| Alternative | Reason rejected |
|-------------|-----------------|
| English only | Limiting for bilingual RFP teams; added i18n cost later is higher than starting clean |
| English + French + Spanish | More complexity than V1 warrants (no Canadian/EU RFP teams confirmed yet) |
| Automatic translation | Quality risk for formal RFP responses; not acceptable for customer-facing exports |
